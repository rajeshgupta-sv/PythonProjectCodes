Multitenancy

Field-Serve as a use case

 

Gap analysis of Straviso readiness towards multitenancy, Field-serv as a use case.

 

Frontend Layer

Customization per tenant

All tenants share same code base, customization per tenant need to be supported through configuration.

Customization of functionality:

In a multitenant Field Serve, code will include all functionality provided to all tenants, but not exposed to all.

Configuration needs to tell which tenant gets which feature. An introduction of a new feature to the code will turned on for tenants who require it and turned off for others

 

Current state:

All changes are deployed to CTL

Future State:

Feature on/off per tenant.

 

Non-functional customization:

Each tenant should be able to customize the look-and-feel for the front end to his preference, such as colors, themes, logo, images, business information etc.

 

Current state:

Field serve customization of CTL front-end is done through Brief-Case.

Future State:

This method could be extended to future tenants.

 

Security

Each tenant should be able to secure the front end to its internal users.

Current state:

CTL uses MDX behind Citrix for security.

Future State:

This method could be extended to future tenants.

 

 

Backend Layer

Customization per tenant

Integration with external systems can be customized per tenant. Payload of external APIs of each tenant should be mapped to the same internal DB structure that is common between all tenants

Current state:

Field serve interacts with CTL through (Gears)

Future State:

How Mapping of new tenants will be standardized/automated??

 

Routing to tenants

Routing: TenantID should flow from the frontend to the backend.

Stateless: In the case of multiple application servers, any application server should be able to process requests of any tenant. The requests are stateless.

Sticky Bit: Serving next request on the same server of previous request has no benefit, as there is no caching.

 

Current state:

No tenantID, no routing.

Future State:

TenantID, Routing layer.

 

Scalability / Load Balancing

Isolating tenants in dedicated servers has a disadvantage of underutilized capacity. Application should support horizontal scalability (adding new server).

�

Current state:

Single server, no load balancer.

Future State:

LB & Horizontal scalability support.

 

Data Layer

Customization per tenant

Data model is common to all tenants, customization is is the level of global (cross-tenant) and internal tenant-specific data.

Data Isolation & Routing

All tenants share the same DB instance. The isolation in the schema level.

(What about schema-less e.g. Mongo)

Security

No query should be able to execute without having the identity of the tenant (carried over from FE)

Scalability

 

Current state:

Vertical scalability only

Future State:

Horizontal scalability.

 

All layers:

Change management/ Code Deployment

Change requests need to be managed in 2 levels, the product level and per-tenant level.

In the product level, all tenants need to be notified of all new releases. In the per-tenant level, the changes need to be transparent to other tenants.

In both cases, deployment of front-end code along with any backend and data changes should be done in a rolling fashion, and eventually deployed to all tenants.

Current state:

Blue-Green (AKA Red-Black or A/B Deployment).

Future State:

Canary deployment, a blue-green in phased approach. With canary deployment, you deploy a new application code in a small part of the production infrastructure. Once the application is signed off for release, only a few users are routed to it. This minimizes any impact.

 

Optional

Future State:

With more tenants, failures get more difficult to track and recover from. Decoupling client and server with message pub-sub/message Q solves the problem.
